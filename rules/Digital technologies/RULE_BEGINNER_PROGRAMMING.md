# Rule: Beginner Programming

## Purpose

Ensure beginner programming resources are clear, scaffolded, curriculum-appropriate and focused on helping students understand how programs work before expecting independent code creation.

## Applies To

- beginner programming tutorials
- Digital Technologies worksheets
- Scratch, micro:bit, Python, JavaScript, GameMaker and web-coding tasks
- debugging activities
- worked examples and answer keys
- teacher notes for programming lessons

## Rule Statement

The agent must teach beginner programming through small, purposeful steps that connect algorithms, code, testing and expected behaviour without overwhelming students with unnecessary syntax or advanced concepts.

## Requirements

- Start with the problem, inputs, outputs and expected behaviour before writing code.
- Use plain language for programming ideas such as sequence, selection, iteration, variable, function, event and input.
- Match the programming environment to the year level, task and user request.
- Prefer visual or block-based examples for earlier learners where appropriate.
- Use short, runnable examples that demonstrate one new idea at a time.
- Move from reading or using existing code, to modifying code, to creating new code when students are novices.
- Show the link between an algorithm or pseudocode and the final program.
- Include testing expectations, such as sample inputs, expected outputs or success checks.
- Explain errors as normal information for improving a program, not as failure.
- Keep code examples copyable and correctly formatted.
- Avoid unnecessary cleverness, advanced syntax, hidden dependencies or unexplained libraries.
- Preserve assessment integrity by giving scaffolds that support thinking without supplying the complete assessed solution.

## Decision Logic

If the learner is new to programming, start with a concrete context, an algorithm and a tiny working program. If the learner already has some experience, ask them to predict, trace, modify or extend code before writing a new solution. If the task is summative, provide structure, testing advice and allowed scaffolds only; do not complete the assessable problem for the student.

## Beginner Progression

1. Define the task in everyday language.
2. Identify inputs, outputs and rules.
3. Write or inspect a simple algorithm.
4. Implement the smallest useful version.
5. Run the program and compare it with expected behaviour.
6. Fix one issue at a time.
7. Add one improvement or extension only after the core version works.

## Examples

- A Scratch task asks students to predict what a sprite will do, run the program, then change one block to alter the movement.
- A Python task introduces `input()`, stores the response in one variable and prints one personalised message before adding branching.
- A debugging activity gives a short program, the intended behaviour and one visible error to investigate.
- A Year 7-8 task uses a simple general-purpose language program with sequence, selection and iteration rather than object-oriented design.

## Non-Examples

- Giving beginners a long full program before they understand the problem it solves.
- Introducing list comprehensions, callbacks or classes in a first lesson unless the user specifically asks for them.
- Asking students to debug without providing expected behaviour.
- Showing code as an image when students need to copy, run or edit it.
- Treating a syntax error as a behaviour problem without helping students read the message.

## Quality Check

- Is the programming idea introduced before extra syntax?
- Is the example short enough for a beginner to trace?
- Are inputs, outputs and expected behaviour clear?
- Can students run or test the code?
- Is the scaffold appropriate for the year level and assessment conditions?
- Has unnecessary advanced content been removed or labelled as extension?
- Is the code formatted so it remains copyable and meaningful?
