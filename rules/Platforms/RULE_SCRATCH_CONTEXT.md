# Rule: Scratch Context

## Purpose

Ensure Scratch tasks are designed for beginner-friendly, visual programming and realistic classroom use.

## Applies To

- Scratch tutorials
- sprite, backdrop and animation tasks
- game-design activities
- debugging tasks
- teacher notes and LMS instructions
- assessment scaffolds using Scratch

## Rule Statement

The agent must treat Scratch as a block-based programming environment for learning computational thinking, not as a general text-code editor.

## Requirements

- Use Scratch vocabulary such as sprite, backdrop, costume, sound, event, block, variable, broadcast and clone.
- Start with visible behaviour students can predict, run, modify and test.
- Explain scripts by event and sprite rather than as one continuous program.
- Use block descriptions or step lists when screenshots are unavailable.
- Keep early tasks focused on sequence, events, loops, decisions, variables and coordinates.
- Account for online versus offline Scratch access.
- Do not assume every student has a Scratch account or can publicly share projects.
- When accounts are used, consider teacher accounts, class studios, comments and project visibility.
- Use appropriate privacy and moderation cautions for student accounts and shared projects.
- Avoid translating Scratch directly into text-code syntax unless the user asks for a bridge to another language.
- Preserve assessment integrity by giving design criteria and debugging prompts rather than complete assessed projects.

## Decision Logic

If the learner is new, use a use-modify-create pattern: inspect an existing script, change one feature, then create a small original feature. If the task involves sharing online, check whether the teacher wants account, studio, comment and visibility guidance. If account access is uncertain, design a task that can work with the offline editor or teacher demonstration.

## Examples

- "When green flag clicked, set score to 0" is described as an event-driven script for the stage or sprite.
- A debugging task asks students to check whether the movement block is inside the repeat loop.
- A game task defines the required sprites, controls, scoring rule and win condition before students build.

## Non-Examples

- Assuming students can create public Scratch accounts without teacher approval.
- Describing Scratch blocks as if they were Python lines.
- Giving a full game solution when the assessment is meant to assess student design.
- Ignoring which sprite owns a script.
- Asking beginners to use clones, lists and custom blocks before the core game works.

## Quality Check

- Is Scratch terminology used accurately?
- Is the task suitable for block-based programming?
- Are accounts, sharing and privacy handled carefully?
- Are sprite ownership and event triggers clear?
- Can students test visible behaviour?
- Is extension separated from the core task?
